## 二、各服务响应时间测量

| 服务 | 端点 | 单次调用耗时 | 调用次数(本文档) | 总耗时估算 |
|------|------|-------------|-----------------|-----------|
| **LLM (Qwen 32B)** | 192.168.11.18:30211 | **~16s/次** | 20 chunks | ~5 分钟 |
| **VLM (Qwen3-VL-8B)** | 192.168.11.18:30254 | **~1s/次** | 332 items | ~6 分钟 |
| **Embedding (bge-m3)** | 192.168.16.36:8081 | **~30ms/次** | ~400 次 | ~12 秒 |
| **Neo4j** | 容器内 | **<5ms** | ~500 次 | ~2 秒 |
| **Qdrant** | 容器内 | **<10ms** | ~400 次 | ~4 秒 |

### 关键发现

1. **LLM 是最大瓶颈**: 单次调用 16 秒，远超其他服务
2. **VLM 单次快但量大**: 332 个 item 串行处理，累计耗时长
3. **Embedding 和向量库不是瓶颈**: 毫秒级响应

## 三、多模态处理详细分析

### 332 个多模态 item 来源

PDF 中提取了 128 张图片 + 11 个图表 + 其他元素。经过 RAG-Anything 上下文增强（每个图片附加周围文本），扩展为 332 个待处理 item。

### 处理流程（每个 item）

```
1. 读取图片文件                    ~1ms
2. 调用 VLM (Qwen3-VL-8B) 生成描述  ~1s
3. 拼接上下文文本                   ~1ms
4. 调用 Embedding 生成向量           ~30ms
5. 写入 Qdrant + Neo4j             ~10ms
                                    ─────
                              合计:  ~1.1s/item
```

332 items × 1.1s ≈ **6 分钟**（纯串行）

## 五、优化建议（按效果排序）

### 1. 并行化多模态处理（预计提速 5-10x）

当前串行处理 332 个 item。改为并发 10-20 个并行请求：

```python
# 当前: 串行
for item in multimodal_items:
    description = await vlm(item)  # 1s each

# 优化: 并发
tasks = [vlm(item) for item in multimodal_items]
descriptions = await asyncio.gather(*tasks, max_concurrency=20)
```

预计多模态阶段从 6 分钟降到 **30-60 秒**。

### 2. 并行化 LightRAG chunk 提取（预计提速 3-5x）

20 个 chunk 串行调 LLM（每次 16s）。改为并发 5 个：

```python
# 当前: 20 × 16s = 320s (5.3 min)
# 优化: ceil(20/5) × 16s = 64s (1 min)
```

### 3. 使用更快的 LLM 模型

当前 Qwen 32B 单次 16 秒。如果换成响应更快的模型（如 Qwen3-8B）：

```
假设 8B 模型 ~2s/次:
20 chunks × 2s = 40s (vs 当前 320s)
```

### 4. 减少 chunk 数量

增大 chunk_size 从 1200 到 2400：
- chunk 数量从 20 减少到 ~10
- LLM 调用减半

### 5. 跳过低价值内容处理

- 关闭 `enable_equation_processing`（方程处理）
- 过滤小于一定尺寸的图片（如 <100x100 像素的图标、logo）
- 跳过 page_header / page_footer 的 VLM 处理

### 6. 缓存机制

- 对相同图片的 VLM 描述结果做哈希缓存
- 已处理的文档支持增量更新
